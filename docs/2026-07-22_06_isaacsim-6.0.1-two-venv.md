# Isaac Sim 6.0.1 — Two-Venv Rollout Architecture

## Background

The rollout Docker stage needs both **openpi** (policy inference, torch 2.7.1
from lockfile) and **Isaac Sim 6.0.1.0** (physics simulation, torch 2.11.0).
They have conflicting torch requirements.

However, openpi and Isaac Sim **never share a Python process** — the policy
runs as a WebSocket server (`backends/openpi/src/openpi/serving/`) and the
Isaac Sim worker connects to it over the network. This means they can use
separate virtual environments with independent torch versions.

## Architecture

```
base (CUDA + apt + uv)
└── deps (uv sync, torch 2.7.1 — openpi + probenet deps)
    ├── trainer (.venv, torch 2.7.1, uv run)
    └── rollout
        ├── .venv           (torch 2.7.1 → policy server)
        └── .venv-isaac     (torch 2.11.0 → Isaac Sim worker)
```

## Why 6.0.1.0 over 5.1.0.0

Isaac Sim 5.1.0.0 had platform packaging issues:
- `isaacsim-core` depends on `pywin32` (Windows-only) — uv resolver rejects it
- `isaacsim-replicator` has no Linux wheel — build fails

Isaac Sim 6.0.1.0 resolved both issues and installs cleanly with `uv pip install`.

## Key Design Decisions

| Decision | Why |
|---|---|
| Two venvs in rollout | openpi policy server (`.venv`, torch 2.7.1) and Isaac Sim worker (`.venv-isaac`, torch 2.11.0) are separate processes — no conflict |
| `--no-deps -e .` instead of `uv sync` in rollout | Sync would reconcile isaacsim's pinned deps (click==8.1.7, psutil==5.9.8) against the lockfile and fail |
| `.venv-isaac/bin/python` for import test | `uv run` would trigger a re-sync and downgrade torch to lockfile's 2.7.1 |
| Single venv in trainer | Lockfile matches — `uv run` works natively |
| Isaac Sim 6.0.1.0 | No pywin32, no replicator wheel issues |
