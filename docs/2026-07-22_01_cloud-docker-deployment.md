# Cloud Docker Deployment

Date: 2026-07-22

## Overview

The project uses **two Docker images** built from a single multi-target Dockerfile,
published to **ghcr.io** via GitHub Actions, and deployed on **Lambda Labs** cloud GPUs.
All coordination between trainer and rollout worker happens through **HuggingFace Hub**
‚Äî no direct networking between instances.

## Why two images

| Backend | torch | Purpose |
|---|---|---|
| openpi (pi0.5) | 2.7.1 | BC fine-tuning |
| Isaac Sim 6.0.1 | 2.11.0 | Simulation & data collection |

The torch version conflict prevents them from coexisting in a single virtual environment,
so two separate Docker targets are used:

| Target | Image tag | GPU | ~Size |
|---|---|---|---|
| `trainer` | `ghcr.io/kindredbluespirit/probenet:main-trainer` | A100 80 GB | 10 GB |
| `rollout` | `ghcr.io/kindredbluespirit/probenet:main-rollout` | A10 24 GB | 30 GB |

GR00T N1.7 (`policies/gr00t/`) requires `torch==2.9.0` and `python<3.13`, which
conflicts with both policy backends. It is not currently included in the Docker images but
could be added as a third target (`:trainer-gr00t`) in the future.

## uv-managed Python (no system Python)

The Docker base image (`nvidia/cuda:12.6.0-cudnn-devel-ubuntu24.04`) does **not**
install Python via apt. Instead:

1. Only `uv` is installed via the official curl script
2. `ENV UV_PYTHON_PREFERENCE=only-managed` ensures uv never falls back to system Python
3. `uv sync --frozen --no-dev` reads `.python-version` (`3.12`) and auto-downloads
   the correct Python version

This makes the image Python-version-independent ‚Äî change `.python-version` and
`uv lock`, rebuild, and uv handles the rest. No `update-alternatives` or
`apt install python3.12*` needed.

## Dockerfile structure

```dockerfile
FROM nvidia/cuda:12.6.0-cudnn-devel-ubuntu24.04 AS base
# apt installs: curl, git, build-essential, GL libs, ffmpeg
# installs uv via curl
# sets UV_PYTHON_PREFERENCE=only-managed

FROM base AS trainer
COPY . .
RUN uv sync --frozen --no-dev
RUN uv run python -c "import openpi; import probenet"
ENTRYPOINT ["uv", "run", "python", "-m", "probenet.cli", "--mode", "trainer"]

FROM base AS rollout
RUN apt-get install -y -qq libxi-dev libglew-dev xorg-dev
COPY . .
RUN uv sync --frozen --no-dev && \
    uv pip install torch==2.11.0 && \
    uv pip install "isaacsim[all,extscache]==6.0.1.0"
RUN uv run python -c "import isaacsim; import probenet"
ENV OMNI_KIT_ACCEPT_EULA=YES
ENTRYPOINT ["uv", "run", "python", "-m", "probenet.cli", "--mode", "rollout"]
```

Key points:
- `COPY . .` copies the entire repo (minus `.git` via `.dockerignore`) into
  `/workspace/probenet/`. The image is fully self-contained ‚Äî configs, scripts,
  submodules, everything.
- `uv sync --frozen --no-dev` bakes the virtual environment at build time.
  No setup at container start.
- The entrypoint directly runs the CLI without any shell wrapper.

## .dockerignore

```
**/.git
```

Everything except `.git` goes into the image. docs, tests, scripts, site/ are
all tiny text files and useful to have inside the container.

## .python-version and uv.lock

- `.python-version` is pinned to `3.12` ‚Äî required for eventual GR00T N1.7 support
  (which pins `requires-python = ">=3.12,<3.13"`), and compatible with both openpi
  and Isaac Sim.
- `uv.lock` is regenerated by running `uv lock` with `.python-version` set. The
  CI build uses `uv sync --frozen` and must match the lock file's Python version
  exactly.

## GitHub Actions ‚Äî docker-publish.yml

Triggered on:
- Push to `main` ‚Üí tags: `main-trainer`, `main-rollout`
- Tags matching `v*` ‚Üí tags: `v1.0-trainer`, `v1.0-rollout`
- Manual `workflow_dispatch`

Uses a matrix build (`trainer`, `rollout`) with `docker/build-push-action@v6`.
Images are published to `ghcr.io/kindredbluespirit/probenet`.

Cache is read-only (`cache-from: type=gha`) ‚Äî the `cache-to` line was removed
because the default `docker` driver doesn't support cache export.

Submodules are checked out recursively since `policies/openpi` is required.

## Lambda Labs deployment

### One-time bootstrap

```bash
curl -sSL https://raw.githubusercontent.com/kindredbluespirit/probenet/main/docker/install_podman.sh | bash
```

Installs podman, NVIDIA container toolkit, and creates data directories.

### Trainer (A100, 80 GB)

```bash
podman run --device nvidia.com/gpu=all \
  -v ~/probenet-data:/data \
  -v ~/probenet-outputs:/workspace/probenet/outputs \
  -e HF_TOKEN=$HF_TOKEN \
  ghcr.io/kindredbluespirit/probenet:main-trainer
```

### Rollout (A10, 24 GB)

```bash
podman run --device nvidia.com/gpu=all \
  -v ~/probenet-data:/data \
  -v ~/probenet-outputs:/workspace/probenet/outputs \
  -v ~/isaac-sim-assets:/isaac-sim/assets:ro \
  -e HF_TOKEN=$HF_TOKEN \
  ghcr.io/kindredbluespirit/probenet:main-rollout
```

No git clone required ‚Äî images are pulled directly from GHCR.

## Distrobox for interactive use

Instead of a separate distrobox image target, use the existing images with an
entrypoint override:

```bash
# On Lambda, after pulling the image:
distrobox create --name probenet-trainer \
  --image ghcr.io/kindredbluespirit/probenet:main-trainer \
  --entrypoint /bin/bash

distrobox enter probenet-trainer
# Inside: uv run python -m probenet.cli --mode trainer
```

Distrobox automatically handles GPU passthrough and filesystem sharing ‚Äî no
manual volume mounts or device flags needed.

## docker-compose.yml (local dev)

For local development on the RTX 3060, `docker/docker-compose.yml` provides
a convenience wrapper:

```bash
podman compose up trainer   # builds and runs trainer locally
podman compose up rollout   # builds and runs rollout locally
podman compose down
```

Uses podman profiles so only one service runs at a time. Builds from the local
Dockerfile, mounts `../data`, `../outputs`, and `../configs` as volumes.
